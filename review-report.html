Critical Issues — Must Fix

#1 Command Injection via Session Names
server.js:85, 91, 98, 108
Security

User-supplied session names are interpolated directly into shell commands with no sanitization. An attacker on the same network can POST to /api/attach with a crafted session name like "; rm -rf /; echo " to execute arbitrary OS commands.

// Current — VULNERABLE
return execSync('tmux capture-pane -e -p -t "' + session + '"');
execSync('tmux send-keys -t "' + session + '" -l ' + escapeShell(keys));
execSync('tmux new-session -d -s "' + name + '" ...');

// Fix — use execFileSync with args array
var execFileSync = require('child_process').execFileSync;
execFileSync('tmux', ['capture-pane', '-e', '-p', '-t', session]);
Insight
Command injection is OWASP's #1 risk. When building shell commands from user input, always use execFile/execFileSync with an args array (which avoids the shell entirely), or validate inputs against a strict allowlist like /^[a-zA-Z0-9_.-]+$/.
#2 Unguarded JSON.parse Crashes the Server
server.js:208, 219, 263, 290
Security Reliability

Every POST handler calls JSON.parse(body) without try/catch. Since there's no process.on('uncaughtException') handler either, a single malformed request kills the entire server.

// Current — will crash on malformed input
var parsed = JSON.parse(body);

// Fix — wrap in try/catch
try {
  var parsed = JSON.parse(body);
} catch(e) {
  res.writeHead(400); res.end('{"error":"Invalid JSON"}'); return;
}
#3 No Request Body Size Limit (Memory DoS)
server.js:321–325
Security

The readBody function accumulates the entire request body into memory with no limit. An attacker could send a multi-gigabyte POST body to exhaust server memory.

// Current — unbounded
req.on('data', function(c) { body += c; });

// Fix — enforce limit
req.on('data', function(c) {
  size += c.length;
  if (size > 64 * 1024) { req.destroy(); return; }
  body += c;
});
#4 XSS via innerHTML with Session Names
server.js:659–661 (client-side JS)
Security

Session names from tmux are injected into the DOM via innerHTML without escaping. A tmux session named <img src=x onerror=alert(1)> would execute JavaScript.

// Current — injects raw session names into HTML
sl.innerHTML = d.sessions.map(function(s) {
  return '<div class="sc-n">' + s.name + '</div>';
}).join("");

// Fix — escape HTML entities
function esc(s) {
  var d = document.createElement('div');
  d.appendChild(document.createTextNode(s));
  return d.innerHTML;
}

Major Issues — Should Fix

#5 Wildcard CORS on a Command Execution Server
server.js:189
Security

Access-Control-Allow-Origin: * means any website open in your browser can send fetch requests to this server and execute tmux commands. A malicious webpage could silently interact with your Claude Code session.

Insight
A server on 0.0.0.0 with CORS: * creates a "browser-as-proxy" attack surface. Any website you visit while this server is running could be a control plane for your terminal. Restrict CORS to the server's own origin or require an auth token.
#6 Synchronous execSync Blocks the Event Loop
server.js:28, 64, 91, 98, 108, 221
Performance

Almost all shell operations use execSync, which blocks the entire Node.js event loop. With a 150ms polling interval calling execSync, every poll blocks all HTTP requests and SSE writes.

Insight
In Node.js, the event loop is single-threaded. execSync freezes all I/O until the child process completes. For a server handling concurrent SSE clients, this means dropped frames and unresponsive endpoints. Use execFile (async) with callbacks.
#7 Unused Native Dependency: node-pty
package.json:10
Maintainability

node-pty is a heavy native module requiring C++ compilation, but v5 doesn't use it anywhere. This adds ~62MB of node_modules, slows installation, and can fail on systems without build tools.

#8 No Graceful Shutdown
server.js (global)
Reliability

There's no SIGTERM/SIGINT handler. On termination, SSE clients receive no notification, the polling interval keeps running, and no cleanup occurs.

1. Modify the readBody function at line 321:

var MAX_BODY = 64 * 1024; // 64KB

function readBody(req, res, cb) {
  var body = '', size = 0;
  req.on('data', function(c) {
    size += c.length;
    if (size > MAX_BODY) {
      res.writeHead(413);
      res.end('{"error":"Body too large"}');
      req.destroy();
      return;
    }
    body += c;
  });
  req.on('end', function() { cb(body); });
}
